# 🚀 OpenShelf v3.2.0 — Database Integrity & Schema Completion

**Release Date:** July 8, 2026

A focused database integrity release that audits the entire `scratch/` migration history against `data/schema.sql` and resolves every discrepancy, ensuring fresh deployments mirror production exactly.

---

## 🌟 What Changed

### 🗄️ Schema Audit & Sync

A full cross-reference was performed between all `scratch/*.php` and `data/migrate*.php` files and the master `data/schema.sql`. Every column, table, and index that existed in production via ad-hoc migrations but was missing from the canonical schema file has now been added.

### 🆕 New Tables Added to Schema

Two financial tables that were previously only created via `scratch/add_support_transactions_tables.php` are now part of the master schema:

#### `support_us`
Stores user donation submissions with full user snapshot, payment provider details, transaction ID, and admin approval tracking.

| Column | Type | Description |
|---|---|---|
| `id` | varchar(30) | Unique submission ID |
| `user_id` | varchar(16) | FK → users |
| `provider` | varchar(50) | bKash / Nagad / Rocket |
| `account_number` | varchar(50) | Sender's account |
| `amount` | decimal(10,2) | Submitted amount |
| `transaction_id` | varchar(100) | Payment TrxID |
| `status` | varchar(20) | pending / approved / rejected |
| `approved_by` | varchar(50) | Admin who approved |

#### `transactions`
Records approved financial transactions linked to a `support_us` entry.

| Column | Type | Description |
|---|---|---|
| `id` | varchar(30) | Unique transaction ID |
| `support_us_id` | varchar(30) | FK → support_us (nullable) |
| `user_id` | varchar(16) | FK → users |
| `invoice_number` | varchar(50) | Generated invoice reference |
| `amount` | decimal(10,2) | Final approved amount |
| `status` | varchar(20) | completed / cancelled |

### 🔑 `idx_return_token` Index

The index on `borrow_requests.return_confirmation_token` — used by the two-step return confirmation flow to quickly look up tokens from owner confirmation emails — is now declared inline in the table definition rather than as a trailing `ALTER TABLE` statement.

### 📋 Schema Cleanup

- The trailing `ALTER TABLE borrow_requests ADD KEY IF NOT EXISTS idx_return_token` statement at the end of the file has been removed since the index is now declared in the table body.
- All previously applied patch migrations are now consolidated into a single clean `schema.sql` — no more running separate migration scripts for a fresh deployment.

## ✅ No Breaking Changes

This release contains no application code changes — only documentation and schema file updates. All existing production databases are unaffected. The changes only impact **fresh deployments** using `data/schema.sql` as the starting point.
